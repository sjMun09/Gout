## Critical

### CRIT-001: SUSPENDED 사용자가 로그인 및 API 접근 가능

- **파일**: `gout-back/src/main/java/com/gout/entity/User.java:83`, `gout-back/src/main/java/com/gout/service/impl/AdminServiceImpl.java:184-198`, `gout-back/src/main/java/com/gout/security/CustomUserDetailsService.java:26`
- **문제**:
  `User.Status` enum 은 `ACTIVE`, `DELETED` 두 값만 정의되어 있다 (`User.java:83`). 그런데 `AdminServiceImpl.updateUserStatus()` 는 JdbcTemplate native SQL 로 `UPDATE users SET status = 'SUSPENDED'` 를 직접 쓴다 (`AdminServiceImpl.java:185`). 그 결과,

  1. **로그인 차단 불가**: `AuthServiceImpl.login()` 은 `user.getStatus() == User.Status.DELETED` 만 검사한다 (`AuthServiceImpl.java:82`). SUSPENDED 상태가 Java enum 에 없어서 `user.getStatus()` 호출 시 Hibernate 가 DB 의 `'SUSPENDED'` 문자열을 역직렬화하지 못해 `IllegalArgumentException` 을 던지거나, enum 직렬화 방식에 따라 `null` 이 된다. 즉 SUSPENDED 유저는 로그인 시 500 에러를 받거나, 드물게 null 비교 분기에서 로그인이 통과될 수 있다.
  2. **JWT 요청 차단 불가**: `CustomUserDetailsService.loadUserByUsername()` 도 DELETED 만 체크한다 (`CustomUserDetailsService.java:26`). SUSPENDED 유저의 기존 액세스 토큰은 만료 전까지 `loadUserByUsername` 호출에서 Hibernate 매핑 오류로 500 이 되거나, 예외가 `UsernameNotFoundException` 으로 wrap 되지 않아 500 이 그대로 전파될 수 있다.
  3. **Refresh 토큰은 차단 가능**: `AuthServiceImpl.refresh()` 는 `user.getStatus() != User.Status.ACTIVE` 로 검사하므로, enum 역직렬화가 `null` 이 되는 경우 에는 `null != ACTIVE` 가 참이 되어 우연히 차단된다. 그러나 동작이 enum 역직렬화 결과에 의존하므로 신뢰할 수 없다.

- **영향**: 관리자가 특정 사용자를 정지해도 해당 사용자의 계정이 실제로 잠기지 않는다. 악성 사용자 정지, 콘텐츠 어뷰저 차단, 계정 도용 대응 등 핵심 관리 기능이 무력화된다.
- **권장 수정**:
  1. `User.Status` enum 에 `SUSPENDED` 를 추가: `public enum Status { ACTIVE, SUSPENDED, DELETED }`
  2. `AdminServiceImpl` 의 native SQL 을 엔티티 메서드로 교체하거나, enum 직렬화와 일치하는 값을 쓰도록 수정
  3. `CustomUserDetailsService.loadUserByUsername()` 에 SUSPENDED 체크 추가: `if (user.getStatus() != User.Status.ACTIVE) throw new UsernameNotFoundException(...)`
  4. `AuthServiceImpl.login()` 에도 ACTIVE 체크 추가
- **연관 이슈**: V23 마이그레이션 관련 TODO 주석 (`AdminServiceImpl.java:193-197`)

## High

### HIGH-001: register/forgot-password 엔드포인트 레이트 리밋 미적용

- **파일**: `gout-back/src/main/java/com/gout/security/RateLimitFilter.java:39-43`, `gout-back/src/main/java/com/gout/controller/AuthController.java:24-25`
- **문제**: `RateLimitFilter` 는 `POST /api/auth/login` 과 `POST /api/posts/{id}/like` 두 경로만 레이트 리밋한다. `POST /api/auth/register` 는 레이트 리밋이 없다. `forgot-password` 엔드포인트는 현재 미구현이지만, 향후 추가 시 동일하게 누락될 가능성이 있다.
- **영향**: 자동화된 계정 대량 생성(계정 폭탄), 이메일 주소 열거(register 시 409 중복 오류를 통해), 이메일 발송 비용 폭발(forgot-password) 등에 노출된다. 특히 register 는 BCrypt 12-cost 연산을 유발하므로 CPU DoS 가능성도 있다.
- **권장 수정**: `RateLimitFilter` 에 `POST /api/auth/register` 경로 추가, IP 기반 버킷 적용 (예: 10 req/min). 향후 forgot-password 추가 시 동일하게 적용.
- **연관 이슈**: P1-9 레이트 리밋 PR

### HIGH-002: Swagger UI 가 운영 환경에서도 공개 접근 가능

- **파일**: `gout-back/src/main/java/com/gout/config/OpenApiWebSecurityCustomizer.java:26-43`
- **문제**: `WebSecurityCustomizer.ignoring()` 으로 `/v3/api-docs/**`, `/swagger-ui/**`, `/webjars/**` 경로를 Spring Security 필터체인에서 완전히 제외한다. 이는 JwtAuthenticationFilter 도 거치지 않음을 의미한다. 주석에 "운영 환경에서 `springdoc.swagger-ui.enabled: false` 로 끄면 된다" 고 적혀 있지만, `application-prod.yml` 에 해당 설정이 없다. 따라서 운영 배포 시 인증 없이 전체 API 명세, 엔드포인트 목록, 파라미터 구조가 공개된다.
- **영향**: 공격자가 Swagger UI 를 통해 존재하는 엔드포인트 전체와 요청/응답 스키마를 파악할 수 있어 공격 표면 정찰이 용이해진다.
- **권장 수정**: `application-prod.yml` 에 `springdoc.swagger-ui.enabled: false` 및 `springdoc.api-docs.enabled: false` 추가. 또는 OpenAPI 경로를 `SecurityConfig.authorizeHttpRequests` 에서 `ADMIN` 역할로 제한.
- **연관 이슈**: 없음

### HIGH-003: post.imageUrls 에 외부 URL 삽입 가능 — Stored SSRF / Open Redirect 잠재 경로

- **파일**: `gout-back/src/main/java/com/gout/dto/request/CreatePostRequest.java:28-29`, `gout-back/src/main/java/com/gout/service/impl/PostServiceImpl.java:160`, `gout-back/src/main/java/com/gout/entity/Post.java:57`
- **문제**: `CreatePostRequest.imageUrls` 는 `List<String>` 이며 Bean Validation 이 전혀 없다. 서비스 계층도 값을 그대로 `Post.imageUrls` 에 저장한다. 프론트엔드는 업로드 후 받은 `/api/uploads/posts/...` 상대 경로를 넣도록 설계되어 있으나, API 를 직접 호출하면 임의 URL (`https://evil.example.com/malware.jpg`, `file:///etc/passwd` 등) 을 저장할 수 있다. 프론트엔드 커뮤니티 상세 페이지는 저장된 URL 을 `<img src={postImageApi.absolute(url)}>` 로 렌더링한다 (`community/[id]/page.tsx:266-270`). `postImageApi.absolute()` 는 `http(s)://` 로 시작하면 그대로 반환한다 (`api.ts:272-276`).
- **영향**:
  - **Stored XSS 경계 약화**: 이미지 태그는 직접 JS 실행은 안 되지만, 악성 서버로의 자동 HTTP 요청이 발생해 트래픽/IP를 수집할 수 있다.
  - **피싱 이미지 삽입**: 신뢰할 수 있는 도메인의 게시글에 외부 이미지를 심어 사용자를 오도한다.
  - **SSRF 준비 단계**: 서버가 나중에 imageUrls 를 fetch 하거나 썸네일을 생성하는 로직이 추가되면 즉시 SSRF 가 된다.
- **권장 수정**: `CreatePostRequest.imageUrls` 에 `@Pattern(regexp = "^/api/uploads/posts/[a-f0-9]{32}\\.(png|jpg|webp)$")` 검증 추가. 서비스 계층에서도 whitelist prefix 검사.
- **연관 이슈**: 없음

### HIGH-004: 다중 인스턴스 배포 시 레이트 리미터 무력화

- **파일**: `gout-back/src/main/java/com/gout/security/RateLimiterService.java:19-22`, `gout-back/src/main/resources/application.yml:47-49`
- **문제**: `RateLimiterService` 가 Bucket4j 상태를 Caffeine in-memory 로만 유지한다. 이미 코드와 TODO 주석에 명시되어 있으나 (`RateLimiterService.java:19-22`), 다중 인스턴스(수평 확장) 배포 시 각 인스턴스가 독립 버킷을 가져 로그인 레이트 리밋이 N배로 느슨해진다. 예를 들어 인스턴스가 3개면 실제 로그인은 5 × 3 = 15 req/min 이 허용된다.
- **영향**: 브루트포스 방어가 수평 확장 환경에서 사실상 무력화된다.
- **권장 수정**: Redis 기반 Bucket4j 백엔드(`bucket4j-redis` 또는 `jcache-redisson`) 도입. Redis 는 이미 refresh token 저장에 사용 중이므로 의존성 추가 없이 전환 가능.
- **연관 이슈**: P1-9 (application.yml TODO 주석 참조)

## Medium

### MED-001: HealthController.currentUserId() — 비인증 시 NullPointerException

- **파일**: `gout-back/src/main/java/com/gout/controller/HealthController.java:95-97`
- **문제**: `currentUserId()` 는 `SecurityContextHolder.getContext().getAuthentication().getName()` 을 체인 호출한다. `authentication` 이 null 이거나 principal 이 `anonymousUser` 인 경우 NPE 가 발생한다. `SecurityConfig` 는 `/api/health/**` 를 명시적으로 `permitAll` 하지 않고 `anyRequest().authenticated()` 로 잡으므로 `JwtAuthenticationFilter` 를 통과한 인증된 요청만 도달한다는 가정이 있다. 그러나 이 가정이 구현 내에서 명시적으로 보장되지 않아 방어 코드가 없다. 다른 컨트롤러(`UserController`, `CommentController`, `BookmarkController` 등)는 모두 null 체크 후 적절한 예외를 던진다.
- **영향**: 필터 체인 변경이나 `@WebMvcTest` 환경에서 NPE → 500 응답 발생 가능.
- **권장 수정**: 다른 컨트롤러와 동일한 패턴으로 null 체크 추가: `Authentication auth = SecurityContextHolder.getContext().getAuthentication(); if (auth == null || !auth.isAuthenticated()) throw new AccessDeniedException(...)`.

### MED-002: 페이지 사이즈 상한 미적용 — 메모리 DoS 잠재

- **파일**: `gout-back/src/main/java/com/gout/controller/NotificationController.java:32`, `gout-back/src/main/java/com/gout/controller/BookmarkController.java:54`, `gout-back/src/main/java/com/gout/service/impl/NotificationServiceImpl.java:29`, `gout-back/src/main/java/com/gout/service/impl/BookmarkServiceImpl.java:60`, `gout-back/src/main/java/com/gout/constant/AppConstants.java:22`
- **문제**: `AppConstants.MAX_PAGE_SIZE = 100` 이 정의되어 있으나 실제로 어떤 서비스/컨트롤러에서도 참조되지 않는다. `NotificationServiceImpl` 과 `BookmarkServiceImpl` 의 pageable 생성은 `size <= 0 ? 20 : size` 만 처리하고 상한을 두지 않아, `?size=10000` 같은 요청이 그대로 DB 쿼리로 전달된다. `PostServiceImpl` 도 동일하다. 비교: `PaperServiceImpl` 은 `Math.min(size, 100)` 으로 올바르게 처리한다 (`PaperServiceImpl.java:32`).
- **영향**: 인증된 사용자 한 명이 `?size=1000000` 요청으로 대량의 데이터를 반환받아 메모리 부족을 유발할 수 있다.
- **권장 수정**: 모든 서비스에서 `Math.min(size <= 0 ? 20 : size, AppConstants.MAX_PAGE_SIZE)` 패턴 적용.
- **연관 이슈**: #52 (PostController 의 trending limit 클램핑은 이미 적용됨, 그러나 일반 목록 API 는 미처리)

### MED-003: 파일 서빙 시 Content-Type 을 filename 확장자로 결정 — MIME Sniffing 위험

- **파일**: `gout-back/src/main/java/com/gout/controller/UploadController.java:142-148`
- **문제**: `serve()` 메서드는 `MediaTypeFactory.getMediaType(filename)` 으로 Content-Type 을 결정한다. 업로드 시에는 `ALLOWED_MIME_TYPES` (image/png, image/jpeg, image/webp) 로 제한하지만, 저장된 파일명의 확장자만 보고 Content-Type 을 결정하므로, 파일명 조작 가능한 경우 `text/html` 을 서빙할 수 있다. 더 중요한 것은 `X-Content-Type-Options: nosniff` 헤더가 응답에 없다는 점이다. Spring Security 의 `contentTypeOptions(Customizer.withDefaults())` 는 `SecurityFilterChain` 이 처리하는 요청에만 헤더를 추가하는데, 이 서빙 엔드포인트는 해당 체인을 통과하므로 실제로는 헤더가 추가된다. 그러나 `Cache-Control: public, max-age=3600` 과 함께 이미지를 캐시하는 구조라 오래된 캐시된 응답에는 헤더가 없을 수 있다.
- **영향**: 구형 브라우저 또는 프록시 캐시 환경에서 MIME 스니핑을 통한 XSS 경로 제공 가능. 실제 가능성은 낮지만 업로드 제한이 올바른지 파일 매직바이트 검증으로 강화 필요.
- **권장 수정**: 업로드 시 파일 매직바이트 검증(`Files.probeContentType` 또는 Apache Tika) 추가. 저장 시 검증된 MIME 을 함께 저장하고 서빙 시 사용.

### MED-004: admin 엔드포인트 권한이 authenticated() + @PreAuthorize 이중 구조 — SUSPENDED 사용자 접근 가능

- **파일**: `gout-back/src/main/java/com/gout/config/SecurityConfig.java:75`, `gout-back/src/main/java/com/gout/controller/AdminUserController.java:27`
- **문제**: `SecurityConfig` 는 `/api/admin/**` 를 `authenticated()` 로만 제어한다 (라인 75). ROLE 검사는 `@PreAuthorize("hasRole('ADMIN')")` 에 위임한다. 이 구조 자체는 올바르다. 그러나 CRIT-001 에서 확인한 것처럼 SUSPENDED 사용자는 기존 JWT 로 여전히 인증 상태를 유지할 수 있다. ADMIN 역할을 가진 사용자가 정지된 경우 해당 사용자의 액세스 토큰 만료 전(최대 15분)까지 관리자 API 를 계속 호출할 수 있다.
- **영향**: CRIT-001 의 파생 이슈. 독립적으로 높은 영향을 주지 않으나, CRIT-001 수정 전까지는 관리자 정지가 즉시 관리자 권한을 회수하지 못한다.
- **권장 수정**: CRIT-001 을 해결하면 동시에 해소됨. 또한 Redis 의 refresh invalidation 은 즉시 적용되지만 access token 은 최대 15분 살아있으므로, 즉각적인 강제 종료가 필요하다면 JTI 블랙리스트(Redis) 도입 검토.

### MED-005: 프론트엔드 관리자 패널 — 클라이언트 사이드 권한 확인

- **파일**: `gout-front/src/app/(main)/admin/layout.tsx:26-38`
- **문제**: 관리자 레이아웃은 `/api/admin/users?size=1` API 를 호출해 200 응답이면 ADMIN 으로 간주한다. 이는 API 탐침을 통한 간접 확인이다. 백엔드가 실제 인가를 수행하므로 보안상 뚫리지는 않지만, 두 가지 부작용이 있다: (1) admin 패널 진입 시 실제 유저 데이터를 읽는 부수 효과 API 호출이 발생한다. (2) 비관리자가 `/admin` URL 을 직접 입력하면 한 번의 불필요한 API 호출 후 리다이렉트된다. JWT 클레임에 `roles` 가 이미 포함되어 있으므로 클라이언트 사이드에서 토큰을 파싱해 역할 확인이 가능하다.
- **영향**: 기능적 문제는 없으나 UX 저하 및 불필요한 API 호출.
- **권장 수정**: `getCurrentUserId()` 패턴처럼 JWT payload 에서 `roles` 클레임을 파싱해 ADMIN 여부를 확인. 백엔드 인가는 이미 올바르게 구성되어 있으므로 이는 UX 개선.

### LOW-002: HealthController 경로가 SecurityConfig permitAll 목록에 없음 — 설계 의도 불명확

- **파일**: `gout-back/src/main/java/com/gout/config/SecurityConfig.java:76-87`, `gout-back/src/main/java/com/gout/controller/HealthController.java`
- **문제**: `SecurityConfig.permitAll` 목록에 `/api/health/**` 가 없다. `anyRequest().authenticated()` 로 보호된다. 이는 올바른 동작이지만, 다른 컨트롤러들처럼 명시적으로 표기하지 않아 의도가 불분명하다. 특히 헬스케어 데이터는 민감 정보(건강 기록 3종)이므로 보호가 당연히 되어야 하는데, 설계 문서 어디에도 이를 명시하지 않는다.
- **영향**: 코드 리뷰 또는 리팩터링 중 실수로 `permitAll` 에 추가될 가능성.
- **권장 수정**: 주석으로 인증 필요 사유 명시, 또는 SecurityConfig 에 명시적 `requestMatchers("/api/health/**").authenticated()` 추가.

### LOW-003: CORS `allowCredentials = true` 이지만 Refresh token 은 LocalStorage 에 저장

- **파일**: `gout-back/src/main/java/com/gout/config/SecurityConfig.java:120`, `gout-front/src/lib/api.ts:47`
- **문제**: CORS 설정에 `allowCredentials = true` 가 있으나, 프론트엔드는 `localStorage.getItem('accessToken')` 으로 토큰을 읽어 Authorization 헤더로 전달한다 (`api.ts:47`). 쿠키 기반 인증을 사용하지 않으므로 `allowCredentials = true` 는 실제로 필요하지 않다(현재 어떤 API 도 `credentials: 'include'` 를 쓰지 않는다). `allowCredentials = true` + 와일드카드 오리진 조합은 CORS 스펙상 금지되어 명시적 오리진 화이트리스트를 강제하는데, 이 부분은 이미 올바르게 구현되어 있다. 다만 미래에 OAuth 쿠키 등을 추가할 때 혼동을 줄 수 있고, 불필요한 Preflight 요청이 발생한다.
- **영향**: 즉각적인 보안 위험은 없으나 설정 의도 불명확.
- **권장 수정**: LocalStorage 기반 JWT 를 계속 사용할 계획이라면 `allowCredentials = false` 로 변경. 향후 쿠키 기반으로 전환한다면 현재 설정은 유지.

## 감사하지 않은 영역

- **OAuth/Kakao 로그인 플로우**: `UserRepository.findByKakaoId()` 가 존재하나 실제 Kakao OAuth 처리 컨트롤러/서비스가 현재 코드베이스에 없음. 향후 구현 시 별도 감사 필요.
- **Frontend 로그인/회원가입 페이지 구현**: `app/(auth)/login/page.tsx`, `register/page.tsx` 가 placeholder stub 만 있어 실제 폼 구현 이후 입력 검증, CSRF 고려사항, 자동완성 attribute 등을 재검토해야 함.
- **DB 마이그레이션 스크립트**: Flyway 마이그레이션 SQL 파일은 이번 감사에서 직접 읽지 않았음. SQL 인젝션 취약점은 JPQL/JdbcTemplate 파라미터 바인딩 확인으로 대체.
- **Redis 접근 보안**: `application-prod.yml` 의 `REDIS_PASSWORD` 가 기본값 없이 선택적으로 설정됨 (`${REDIS_PASSWORD:}`). 비밀번호 미설정 시 인증 없이 Redis 접근 가능 — 네트워크 경계로 보호되는지 확인 필요.
- **Docker Nginx TLS 설정**: `docker-compose.prod.yml` 에 nginx 컨테이너가 443 포트를 열고 SSL 볼륨을 마운트하나, `docker/nginx.conf` 는 감사 범위에 포함하지 않음.
- **논문 크롤러 AI 연동**: `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` 사용 코드(`PaperAiService`, `PaperEmbeddingService`)는 외부 API 호출 및 비용 폭발 가능성이 있으나 `PAPER_CRAWLER_ENABLED=false` 기본값으로 현재는 비활성. 활성화 전 별도 검토 필요.
